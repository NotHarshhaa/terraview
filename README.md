## What is Terraview?

Terraview reads your Terraform project ‚ÄĒ `.tf` files, state backend, and optionally a JSON plan ‚ÄĒ then renders a **live status grid** grouped by cloud provider and service type. No SaaS account required: run a single binary locally, in Docker, or behind your CI pipeline.

**Backend:** Go engine + HTTP API + background poller  
**Frontend:** Next.js dashboard (shadcn / radix-sera preset)

---

## Why Terraview?

| Pain | Terraview's answer |
|---|---|
| `terraform state list` is a flat text dump | Visual grid grouped by provider ‚Äļ service (or module) |
| GUI tools are often SaaS or enterprise-only | Self-hosted binary; optional basic auth |
| Hard to see pending vs applied at a glance | Eight lifecycle statuses with filters and summary chips |
| Drift only visible after `terraform plan` | Plan JSON `resource_drift` surfaced as **drifted** status |
| Sharing infra status with non-engineers | Shareable filter URLs; export JSON/CSV |

## Features

### Engine & backends

- **Zero-config local mode** ‚ÄĒ point at a directory; discovers `.tf` files and `terraform.tfstate`
- **Multi-backend state** ‚ÄĒ local, S3, GCS, Azure Blob, Terraform Cloud / HCP Terraform
- **Plan ingestion** ‚ÄĒ optional `plan_file` (`terraform show -json`) for pending changes and drift
- **Plan & drift metadata** ‚ÄĒ `plan_action` and `drift_attributes` on each resource when a plan is loaded
- **State metadata** ‚ÄĒ `state_serial` and `state_modified_at` on snapshots (local backend mtime)
- **Eight lifecycle statuses** ‚ÄĒ `created`, `inactive`, `pending_create`, `pending_update`, `pending_destroy`, `drifted`, `unmanaged`, `unknown`
- **Auto-categorization** ‚ÄĒ AWS / GCP / Azure / Kubernetes ‚Üí Compute, Networking, Databases, Storage, IAM, Serverless, ‚Ä¶
- **Module-aware** ‚ÄĒ shows module path per resource
- **Live polling** ‚ÄĒ background refresh (default 30s) + SSE push to the UI
- **CI mode** ‚ÄĒ `terraview status` prints JSON or Markdown; exit code `2` when drift is detected

### Dashboard (UI)

- **Filter sidebar** ‚ÄĒ search, provider, category, module, and tag facets
- **Quick filters** ‚ÄĒ Needs attention, Drifted, Pending changes, Healthy, Unmanaged
- **Saved views** ‚ÄĒ persist named filter sets in the browser
- **Shareable URLs** ‚ÄĒ filters sync to query params (`?status=drifted&provider=AWS`)
- **Summary bar + status distribution** ‚ÄĒ clickable status chips and segment bar
- **Attention banner** ‚ÄĒ highlights resources that need action
- **Group by service or module** ‚ÄĒ toggle grid grouping; preference saved locally
- **Sort & density** ‚ÄĒ sort by name, status, type, or address; compact row mode
- **Collapsible groups** ‚ÄĒ expand/collapse all resource sections
- **Resource detail sheet** ‚ÄĒ full metadata, tags, copy address, Terraform CLI hints, plan action & drift attributes
- **Provider breakdown chart** ‚ÄĒ clickable bar chart by cloud provider
- **State info bar** ‚ÄĒ state serial and last-modified timestamp from backend
- **CI headline in header** ‚ÄĒ live status summary from `/api/status`
- **Tag filter from detail** ‚ÄĒ click a tag in the detail sheet to filter the grid
- **Markdown export** ‚ÄĒ download filtered resources as a Markdown report
- **Cloud service icons** ‚ÄĒ official AWS/GCP/Azure SVGs via `@thesvg/cli` (group headers, rows, provider chips)
- **Deep links** ‚ÄĒ `#resource=aws_instance.web` opens the detail panel
- **Command palette** ‚ÄĒ `Ctrl+K` / `‚ĆėK` to jump to any resource
- **Keyboard shortcuts** ‚ÄĒ `/` search, `r` refresh, `Esc` clear filters, `?` help
- **Export** ‚ÄĒ download filtered resources as JSON or CSV; copy view link
- **Live connection badge** ‚ÄĒ Live / Polling / Offline SSE status in the header
- **Theme toggle** ‚ÄĒ light, dark, or system
- **Optional auth UI** ‚ÄĒ login form when basic auth is enabled

### API

| Endpoint | Description |
|---|---|
| `GET /api/health` | Liveness + version |
| `GET /api/snapshot` | Full snapshot (resources, summary, UI config) |
| `GET /api/resources` | Filtered resource list (`?status=&provider=&module=&category=&tag=&q=&limit=&offset=`) |
| `GET /api/resource` | Single resource by address (`?address=aws_instance.web`) |
| `GET /api/facets` | Filter facet counts (optionally pre-filtered) |
| `GET /api/summary` | Aggregate counts only |
| `GET /api/status` | Compact headline for badges / CI |
| `POST /api/refresh` | Force refresh |
| `GET /api/events` | SSE stream (`refreshed` events) |
| `POST /api/login` | Exchange credentials for session token (when auth enabled) |

## Quick start

### Prerequisites

- Go **1.25+** (to build from source)
- Node **20+** (UI development only)

### Binary

```bash
git clone https://github.com/NotHarshhaa/terraview
cd terraview

go run ./cmd/terraview serve ./testdata/sample-project
# API + UI (if ui/out exists) ‚Üí http://localhost:7777
```

### Development (API + UI)

Use two terminals ‚ÄĒ the UI proxies `/api/*` to the Go server via Next.js rewrites (no CORS setup needed):

```bash
# Terminal 1 ‚ÄĒ API
go run ./cmd/terraview serve ./testdata/sample-project --no-ui

# Terminal 2 ‚ÄĒ UI
cd ui && npm install && npm run dev
# ‚Üí http://localhost:3000
```

Or use the Makefile:

```bash
make run          # build + serve sample project on :7777
make ui-dev       # Next.js on :3000 (requires API on :7777)
make test         # go test ./...
```

### Docker

```bash
docker compose up --build
# or pull the published image:
docker pull ghcr.io/notharshhaa/terraview:latest
docker run -p 7777:7777 -v "$(pwd):/workspace" ghcr.io/notharshhaa/terraview:latest
```

Images are built and published to [GHCR](https://github.com/NotHarshhaa/terraview/pkgs/container/terraview) when you:

- **Create a version tag** ‚ÄĒ e.g. `git tag v0.1.0 && git push origin v0.1.0` (publishes `v0.1.0`, `0.1.0`, `0.1`, and `latest`)
- **Run the workflow manually** ‚ÄĒ Actions ‚Üí *Publish Docker image* ‚Üí *Run workflow* (publishes `sha-<commit>`; optionally check *Also tag as latest*)

### CI / PR comments

Use the published **[Terraview Status Check](https://github.com/NotHarshhaa/terraview-action)** GitHub Action (Marketplace) or the CLI directly.

**GitHub Actions:**

```yaml
permissions:
  contents: read
  pull-requests: write

jobs:
  terraview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: NotHarshhaa/terraview-action@v1
        with:
          working-directory: ./infra
          mode: status-check          # also: drift-gate, destroy-guard, summary-report
          plan-file: ./plan.json
          # backend: s3
```

| Mode | Behavior |
|---|---|
| `status-check` | Post/update PR comment with resource table (read-only) |
| `drift-gate` | Fail when drift is detected (exit 2) |
| `destroy-guard` | Fail when any `pending_destroy` resources exist |
| `summary-report` | Write JSON/HTML report files for pipeline artifacts |

See the [terraview-action README](https://github.com/NotHarshhaa/terraview-action) for full inputs and workflow examples.

**CLI-only CI** (no Action wrapper):

```bash
terraview status ./infra --format markdown
terraview status ./infra --plan-file ./plan.json   # includes pending + drift
# Exit 2 if any resource is drifted
```

See [`.terraview.yaml.example`](.terraview.yaml.example).

## Status classification

Each resource is classified from HCL declarations, state, and plan:

```
In state?
‚Ēú‚ĒÄ‚ĒÄ NO  ‚Üí in plan as create?  ‚Üí pending_create
‚Ēā         declared in .tf?    ‚Üí unmanaged
‚Ēā         else                ‚Üí unknown
‚ĒĒ‚ĒÄ‚ĒÄ YES ‚Üí in plan?
‚Ēā         ‚Ēú‚ĒÄ‚ĒÄ delete          ‚Üí pending_destroy
‚Ēā         ‚Ēú‚ĒÄ‚ĒÄ update/replace  ‚Üí pending_update
‚Ēā         ‚ĒĒ‚ĒÄ‚ĒÄ create          ‚Üí pending_create
          drift in plan?        ‚Üí drifted
          provider inactive?    ‚Üí inactive
          else                  ‚Üí created
```

**Drift detection** reads the `resource_drift` section from a Terraform plan JSON file. Pass it via `plan_file` in config or `--plan-file` on the CLI.

## Configuration

Copy [`.terraview.yaml.example`](.terraview.yaml.example) to your project root. All fields are optional.

```yaml
port: 7777
poll_interval: 30s
working_dir: .
plan_file: ./plan.json          # optional: terraform show -json output

backend:
  type: local                   # local | s3 | gcs | azureblob | tfc
  # S3: bucket, key, region, dynamodb_table, endpoint
  # GCS: bucket, key
  # Azure: storage_account, container, key
  # TFC: organization, workspace, token, hostname

ui:
  title: "My Project ‚ÄĒ Infrastructure"
  show_cost_column: false       # reserved for future Infracost integration
  default_filter: status=created

auth:
  enabled: false
  username: admin
  password_env: TV_PASSWORD
  access_token: secret-token    # Bearer / ?access_token= for SSE
```

### Environment variables

| Variable | Default | Description |
|---|---|---|
| `TV_PORT` | `7777` | HTTP port |
| `TV_POLL_INTERVAL` | `30s` | Snapshot refresh interval (min 5s) |
| `TV_WORKING_DIR` | `.` | Terraform project root |
| `TV_BACKEND` | `local` | Backend type |
| `TV_STATE_BUCKET` | ‚ÄĒ | S3/GCS bucket |
| `TV_STATE_KEY` | ‚ÄĒ | State object key |
| `TV_STATE_REGION` | ‚ÄĒ | AWS region (S3) |
| `TV_STATE_FILE` | ‚ÄĒ | Explicit local state path |
| `TV_PLAN_FILE` | ‚ÄĒ | Plan JSON path |
| `TV_UI_TITLE` | `Terraview` | Dashboard title |
| `TV_PASSWORD` | ‚ÄĒ | Basic auth password |
| `TV_ACCESS_TOKEN` | ‚ÄĒ | Static API token (SSE-friendly) |
| `TFE_TOKEN` | ‚ÄĒ | Terraform Cloud token |

### Auth

When `auth.enabled: true`, the API accepts:

- HTTP **Basic** auth (`username` / `password`)
- **Bearer** token header (`access_token` or session token from login)
- **`?access_token=`** query param (required for browser EventSource / SSE)
- **Session cookie** from `POST /api/login`

The UI stores credentials in `sessionStorage` and shows a login form on `401`.
